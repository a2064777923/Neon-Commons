# 後台使用說明

## 入口與邊界

- 後台入口：`/admin`
- 需要管理員 session；非管理員只會拿到權限錯誤
- Live 房間資料來自 backend runtime，不需要直接查 PostgreSQL 才能看房況
- 這一版後台維持 split runtime：frontend 在 `3100`，backend 在 `3101`

## 遊戲家族開關

後台首頁先顯示 card / party / board 三個遊戲家族的能力開關。這一區的原則是「只影響新房」：

- 停用某個遊戲，只會阻擋新的建房入口
- 已經存在的房間不會被原地改寫
- 每次切換都會在「最近變更」留下對應留痕

如果你只是要短暫停止新房建立，優先用家族開關，不要直接去動 live 房間。

## Live 房間操作

這是 v1.1 新增的主要營運面。操作順序是先看左側 live 房間目錄，再用右側詳情面板做介入。

### 目錄會顯示什麼

- 房號、遊戲家族、房間標題
- 房內真人 / 總人數
- 最後活動時間
- `live` 或 `snapshot-only` 狀態

`snapshot-only` 代表入口快照還在，但當前已沒有可操作的 live runtime。這種房間可以看恢復資訊，但不應做破壞性操作。

### 主要操作

- `查看房間詳情`
  讀取該房的 config 摘要、runtime health、房內身份與 presence 狀態。
- `排空房間`
  阻止新加入，讓現有玩家走完離場或重連流程。需要再次輸入房號確認。
- `關閉房間`
  立即終止這個 live 房間。需要輸入 `CLOSE` 確認。
- `移除玩家`
  對可移除的房內身份做明確踢出。遊客被移除後，必須重新透過有效邀請入場。

### 詳情面板重點

- `真人`、`AI`、`連線中` 三個 runtime health 指標
- 房間 config 摘要，例如模板、可見性、局別或棋局設定
- 房內身份列表，包含座位、是否房主、是否 AI、`connected / reconnecting / disconnected` presence

### 留痕約定

每次 `查看房間詳情`、`排空房間`、`關閉房間`、`移除玩家` 都應該在首頁底部的「最近變更」追加一筆 audit row。做 live 房間介入後，先回看這個 feed，確認：

- 動作類型正確
- 房號正確
- 操作原因與時間合理

## Runtime Controls 與模板

後台仍保留兩類基礎營運配置：

- Runtime controls
  例如 `maxOpenRoomsPerUser`、`maintenanceMode`
- 模板管理
  用來維護規則模板與對應的房間摘要輸出

注意事項：

- `maintenanceMode` 打開後會阻擋新房建立，但不會直接踢掉現有房間
- 模板更新後，新建房的規則摘要要和模板卡片保持一致；如果不一致，先回頭檢查模板 payload 正規化

## 玩家管理

玩家管理仍放在首頁主視圖，不藏進二級頁：

- 加虛擬豆
- 加段位分
- 封禁 / 解封

這些操作同樣會回寫「最近變更」，方便把房間介入和玩家介入放在同一條審計時間線上看。

## 建議的營運節奏

1. 先用家族開關或 runtime controls 處理全域新房策略。
2. 只在需要針對單房介入時，再進入 live 房間目錄。
3. 先 `查看房間詳情`，確認是 live room 還是 `snapshot-only`。
4. 需要保守收斂時用 `排空房間`；確定要立即終止時才用 `關閉房間`。
5. 單一身份問題才用 `移除玩家`。
6. 每次操作後回看「最近變更」，確認留痕完整。
