# Phase 11 Plan 01: Define shared availability and degraded-mode contracts

<objective>
Create one backend-owned, brownfield-safe availability contract that separates room-state truth from subsystem degradation.

Purpose: Phase 9 and Phase 10 already established truthful room availability and release verification, but the repo still has no shared way to represent degraded entry, realtime, or voice conditions. This plan introduces the canonical enums, normalization, and additive payload fields that later runtime controls and UI messaging will consume.
Output: a shared availability helper module, normalized degraded-mode state on the admin runtime surface, and additive hub / room-entry / room-detail payload fields that all speak the same language without breaking current routes.
</objective>

<threat_model>
<trust_boundaries>
- Availability state crosses the backend/frontend boundary; if the contract is ambiguous, different pages will render contradictory degraded behavior.
- The admin runtime surface is an operator-facing source of truth, so unsafe defaults or malformed degraded-mode config would directly affect live player flows.
- Recovery-aware `409` payloads and normal `200` room payloads both feed the entry and room experience; drift between them would make degraded behavior inconsistent or misleading.
</trust_boundaries>
