# Phase 9 Plan 02: Add stale-room expiry and abandonment cleanup rules

<objective>
Introduce one predictable abandonment lifecycle for live rooms and persisted snapshots.

Purpose: Restart-safe discovery is only useful if stale rooms eventually disappear. Phase 9 therefore needs a single-node cleanup model that respects reconnect grace and clears both in-memory room state and persisted directory rows together.
Output: shared expiry-policy helpers, manager-level abandonment cleanup, and node tests proving stale rooms disappear predictably across card, party, and board families.
</objective>

<threat_model>
<trust_boundaries>
- Reconnect grace and stale cleanup share ownership across three room managers; ordering bugs can prematurely evict valid seats or leave ghost rooms behind.
- Expiry policy crosses the boundary from persistent config into runtime timers; invalid coercion could create effectively infinite or zero cleanup windows.
- Snapshot prune logic touches both runtime memory and database rows; one-sided cleanup would leak stale discovery state.
</trust_boundaries>
