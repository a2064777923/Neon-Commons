**Status:** SHIPPED 2026-04-22

## Overview

v1.0 turned Hong's Neon-Commons from an already-live party arcade baseline into a better-defined, better-verified shipped platform. The milestone hardened the split frontend/backend contract, expanded the admin control plane, unified hub and room-entry flows, shipped the first new-game wave, deepened card/party and board gameplay, and ended with a canonical deployed-stack release gate on `3100/3101`.

## Key Accomplishments

- Hardened the dedicated backend contract and shared frontend network runtime so browser pages, tests, and handlers all point at one consistent API/socket boundary.
- Expanded the admin control plane with capability toggles, runtime controls, audit logs, and richer template operations.
- Unified the hub, room-number resolution, invite deep links, and guest-entry flow across shipped game families.
- Shipped Wave 1 titles on top of the shared platform model: Sokoban, Reversi, and Undercover.
- Expanded live Dou Dizhu, Werewolf, Avalon, Gomoku, and Chinese Checkers behavior without regressing the shared room framework.
- Added a canonical release routine with `deploy:3100`, `test:logic:critical`, `test:ui:critical`, and `verify:release`.

## Milestone Summary

**Key Decisions**

- Kept the separated frontend/backend runtime in one repository and treated that boundary as the baseline contract.
- Promoted shared room-directory and room-entry resolution as the universal cross-family room model.
- Treated the deployed `3100/3101` stack as the canonical verification target instead of ad hoc debug ports.

**Issues Resolved**

- Removed drift between backend route ownership and frontend callers.
- Removed admin-control-plane dependence on direct DB edits.
- Removed fragmented release verification commands by consolidating them into stable scripts and one release runner.

**Issues Deferred**

- Distributed room-state recovery beyond a single in-memory node
- TURN / SFU infrastructure for party-room voice
- Native mobile clients
- Richer economy and monetization systems

**Technical Debt**

- No standalone `v1.0-MILESTONE-AUDIT.md` existed at close; milestone confidence came from phase summaries, requirement traceability, open-artifact audit, and Phase 6 release verification.
- The current repo still carries a large uncommitted working-tree delta at milestone close, so git history prior to this close does not reflect the full v1.0 feature arc phase by phase.
