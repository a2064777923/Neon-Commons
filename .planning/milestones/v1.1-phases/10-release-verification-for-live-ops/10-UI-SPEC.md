# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope

- Preserve the shipped split visual language instead of introducing a new system: dark neon arcade for public hub discovery, warm utility console styling for entry/admin workflows.
- Define smoke-stable recovery states for the hub live feed, room entry gate, and recoverable room UIs so `verify:release` can catch regressions before ship.
- Define the future live-room-ops admin workflow in the same visual grammar already used on `/admin`, with explicit selectors and copy that Playwright can target.
- Keep Tool = `none`; this brownfield Next.js + CSS modules project should extend existing modules and shared global classes only.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon separation, tight badge offsets |
| sm | 8px | Label-to-value gaps, stacked status copy, inline control clusters |
| md | 16px | Default form spacing, card internals, row padding |
| lg | 24px | Panel padding, two-column content gaps, section headers |
| xl | 32px | Major card padding, hero-to-panel separation |
| 2xl | 48px | Page section breaks, admin split-panel breathing room |
| 3xl | 64px | Top-level page rhythm on desktop |

Exceptions: shared action controls may keep the existing `46px` minimum height from `styles/globals.css`. Legacy non-scale paddings already present in untouched brownfield files are grandfathered and should be treated as migration debt, not as new Phase 10 tokens. Any new Phase 10-specific layout gap or padding introduced in touched code must snap to the scale above.

Typography notes:

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#050913` on hub, `#fffbf2` on entry/admin | Root page background and the visual tone of the current surface mode |
| Secondary (30%) | `rgba(8,18,33,0.76)` on hub, `rgba(255,255,255,0.72)` on entry/admin | Cards, side rails, room detail panels, audit containers |
| Accent (10%) | `#67ebff` on hub, `#234033` on entry/admin | Primary actions, selected rows, status emphasis, focus-forward affordances only |
| Destructive | `#ff6b8b` | Close room, drain room, remove occupant, hard failure text |

Accent reserved for: primary CTA buttons, active mode tabs, selected live-room row highlight, recovery status badges, room-availability chips, inline confirmation focus states, and top-of-feed operator success emphasis. Do not use accent for long-form paragraph text, full-page backgrounds, or non-interactive decorative blocks.

Supporting highlight contract:

- `#ffb454` / existing warm gold may be used for timestamp, “last activity,” and recovery-adjacent metadata chips.
- `#72ffca` remains success-only for confirmation or post-action completion states.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | 查看房間詳情 |
| Empty state heading | 目前沒有 live 房間 |
| Empty state body | 還沒有可檢查的房間。等待玩家開房，或先建立一個測試房後再重新整理。 |
| Error state | 房間資料同步失敗。請重新整理控制台；若仍失敗，先確認 `3100/3101` 服務已就緒，再重跑發布驗證。 |
| Destructive confirmation | 排空房間: 排空後會阻止新加入，並讓現有玩家走完離場流程。輸入房號確認。; 關閉房間: 關閉會立即終止這個 live 房。輸入 `CLOSE` 確認。; 移除玩家: 移除後該身份會離開目前房間；遊客必須重新透過有效邀請入場。再次點擊「移除玩家」確認。 |

Copy tone contract:

- Use zh-Hant operational copy, matching the shipped hub and admin surfaces.
- Recovery copy must explain whether the room is visible, joinable, or blocked right now. Do not replace those distinctions with vague “稍後再試”.
- Admin action labels stay verb-first and concrete: `查看房間詳情`, `排空房間`, `關閉房間`, `移除玩家`.
- Keep English eyebrow labels already present in the brownfield UI (`ROOM ENTRY`, `Admin Console`, `NEON COMMONS`), but stateful operator copy and smoke assertions should anchor to the Chinese action/status text.

## Interaction Contract

### Recovery-Aware Hub And Entry

| Surface | Required state | Contract |
|--------|----------------|----------|
| Hub live feed row | `live` | Row remains clickable and routes through `room.entryRoute || room.sharePath || room.detailRoute`; status label stays `待開局` or `對局中`. |
| Hub live feed row | `snapshot-only` | Row stays visible in discovery, uses the recovery visual variant, appends `· 重啟恢復中` in supporting copy, and shows the compact status label `恢復中`. |
| Share-room selector option | `snapshot-only` | Option text appends `· 恢復中`; copying the invite stays allowed so operators can preserve the route during recovery. |
| Entry page | `loading` | Show the existing utility hero and status rail immediately; avoid blank states or skeleton-only rendering. |
| Entry page | `snapshot-only` | Status block must show `重啟恢復中`; method copy must say `恢復快照可見，但暫停進場`; guest/login buttons stay disabled with labels `房間恢復中` and `等待房間恢復`; keep the return link visible. |
| Entry page | recoverable user | Preserve automatic return behavior and the notice `已檢測到可恢復的登入身份，系統會自動帶你回到房內。` |
| Entry page | recoverable guest | Preserve automatic guest recovery and the notice `你已經持有這個房間的遊客身份，正在恢復這個席位。` |
| Entry page | not found / failed resolve | Show inline error text in the existing error treatment, while keeping the page shell and recovery notes visible. |

### Future Live-Room-Ops Admin Workflows

| Workflow | Contract |
|---------|----------|
| Live room directory | Add a first-class panel on `/admin` using the current `UtilityPages.module.css` panel grammar. Each row shows family, game title, room number, occupancy, availability, and last activity in one scan line. |
| Room selection | Selecting a room updates an inline detail panel on the same page. Do not navigate away or open a full-screen modal for the default inspect action. |
| Room detail | Detail view shows config summary, runtime health, occupant list, and last activity with the same heading/body scale already used in admin panels. |
| Drain room | Action is destructive-but-recoverable. Use a secondary confirmation step inline in the detail panel, not a browser confirm dialog. |
| Close room | Action is destructive and immediate. Use the destructive color token and the strictest confirmation copy in this phase. |
| Remove occupant | Action belongs inside the occupant row in the detail panel, with the same audit-after-action behavior as other admin controls. |
| Audit feedback | After any room-op action, prepend a new audit row to the existing “最近變更” feed instead of inventing a second history widget. |

Primary visual anchor and scan order for the live-room-ops admin surface:

1. The live room directory panel is the first-read anchor on desktop and tablet; it should dominate the initial scan with occupancy, availability, and last-activity information.
2. The selected live-room row becomes the local focal point through accent treatment and inline status emphasis, not through a route change.
3. The room detail panel is the second-read area: config, runtime health, and occupants deepen the currently selected room without competing with the directory for initial attention.
4. Destructive controls read last and stay visually subordinate until a room is selected and a confirm state is opened.

## Verification Hooks

### Existing hooks that must remain unchanged

| Hook | Surface | Purpose |
|------|---------|---------|
| `data-presence-state` | room UIs | Reconnect/disconnect smoke coverage across shipped room families |
| `data-recovery-state` | room UIs | Recovery banner and reconnect affordance coverage |
| `data-admin-family` | admin | Existing grouped family capability smoke |
| `data-game-key` | admin | Existing per-game toggle smoke |
| `data-audit-row` | admin | Existing audit feed assertions |
| `data-admin-template` | admin | Existing template editor smoke |

### New hooks required for Phase 8 + Phase 10 coverage

| Hook | Surface | Contract |
|------|---------|----------|
| `data-live-feed-room="{roomNo}"` | hub | Attach to each live feed card so snapshot-only discovery can be asserted without relying on marketing copy. |
| `data-room-availability="{live|snapshot-only|draining|closed}"` | hub, entry, admin | Use the same enum everywhere the room state is surfaced. |
| `data-entry-action="{guest|login}"` | entry | Preserve disabled/enabled state assertions for blocked recovery entry. |
| `data-entry-notice="{recovery|guest-recovery|snapshot-only}"` | entry | Make release smoke resilient to copy refinements while keeping the message category explicit. |
| `data-live-room-directory="true"` | admin | Directory shell for future room-ops smoke. |
| `data-live-room-row="{roomNo}"` | admin | Stable selector for inspect and post-action status assertions. |
| `data-live-room-detail="{roomNo}"` | admin | Selected room detail shell. |
| `data-room-action="{inspect|drain|close}"` | admin | Stable action buttons. |
| `data-room-action-confirm="{drain|close|remove-occupant}"` | admin | Inline confirm state hook. |
| `data-room-occupant="{occupantId}"` | admin | Occupant row selection and presence assertions. |
| `data-room-occupant-action="remove"` | admin | Remove-occupant action hook. |

## Release-Gate Visual Preservation

- Do not swap the hub to the warm utility theme; recovery discovery remains part of the neon arcade home surface.
- Do not move room recovery messaging into toast-only UI. Snapshot-only and recoverable states must remain visible inline on the entry page.
- Do not hide live-room interventions behind JSON editors or freeform textareas. They belong in explicit admin controls consistent with the existing runtime/capability panels.
- Do not invent a separate audit history surface for room ops. Phase 10 assumes one operator can verify intervention results in the existing audit feed.
- Do not require shell access or database knowledge to understand a room’s state. The admin detail panel must expose config, occupants, runtime health, and last activity in one glanceable view.
