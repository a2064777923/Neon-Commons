# Milestone v1.1: Live Ops & Reliability

**Status:** SHIPPED 2026-04-23
**Phases:** 7-10
**Total Plans:** 9

## Overview

v1.1 turned Hong's Neon-Commons from a broad but mostly happy-path real-time arcade into a recovery-aware and operator-ready platform. The milestone shipped reconnect continuity across the room families, added honest single-node restart behavior through snapshot-only discovery and stale-room expiry, brought live-room inspection and intervention into the admin surface, and closed with a canonical release gate that proves the new live-ops and recovery contract on the deployed `3100/3101` stack.

## Key Accomplishments

- Shipped refresh/reconnect recovery and host-visible presence across Dou Dizhu, party rooms, Undercover, board rooms, and Reversi.
- Added single-node room-directory snapshot persistence, stale-room expiry, and `snapshot-only` discovery so restart recovery stays visible without pretending rooms are live.
- Delivered the minimum truthful live-room operations surface: admin room directory, room detail, drain, close, and occupant removal flows with audit coverage.
- Expanded `verify:release` and the critical node/browser suites so reconnect, stale-room cleanup, and operator interventions are now release blockers instead of tribal-knowledge checks.
- Refreshed operator docs, roadmap status, and state tracking so the shipped v1.1 contract matches the artifacts on disk.

## Milestone Summary

**Key Decisions**

- Prioritized live room operations and single-node reliability before another new-content wave so the shipped game surface could be operated safely.
- Absorbed the minimum truthful Phase 8 room-ops surface into Phase 10-01 because release verification could not honestly prove operator interventions against nonexistent UI/API paths.
- Kept `npm run verify:release` as the single canonical pre-ship command, with narrow live-ops reruns added only for diagnosis.

**Issues Resolved**

- Resolved refresh/reconnect seat loss by standardizing recovery metadata and bounded reconnect grace across room families.
- Resolved single-node restart ambiguity by exposing explicit `live` and `snapshot-only` availability instead of ghost-room behavior.
- Resolved the lack of targeted admin intervention by shipping live-room inspect, drain, close, and occupant removal workflows.
- Resolved vague release diagnostics by naming the live-ops and recovery logic/UI gates explicitly and documenting the operator flow.

**Issues Deferred**

- Distributed room-state recovery beyond the current single-node model
- TURN / SFU infrastructure for party-room voice
- Wave 2 new-game expansion after the live-ops baseline
- Native mobile clients
- Richer economy and monetization systems

**Technical Debt**

- No standalone `v1.1-MILESTONE-AUDIT.md` existed at close; milestone confidence came from phase summaries, open-artifact audit, and the green canonical release gate.
- Phase 8 was delivered inline during Phase 10 execution, so the original roadmap ordering is preserved only as traceability notes rather than standalone phase execution history.
- Runtime recovery still stops at honest single-node visibility; it does not yet provide multi-node room takeover or state migration.
