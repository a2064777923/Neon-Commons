# Phase 03.1: New Game Delivery Wave 1 Summary

**Undercover party-room gameplay, dedicated `/undercover/{roomNo}` UX, and a full Wave 1 regression gate across new and shipped titles**

## Performance

- **Duration:** 26 min
- **Started:** 2026-04-22T15:24:31+08:00
- **Completed:** 2026-04-22T15:50:31+08:00
- **Tasks:** 2
- **Files modified:** 8

## Accomplishments

- Added a pure Undercover round engine with word-pair assignment, ordered clue turns, vote elimination, reveal state, and win conditions.
- Integrated Undercover into the shared party manager and shipped a dedicated `/undercover/{roomNo}` room page with clue entry, voting, guest-safe routing, and result flow.
- Closed Wave 1 with consolidated verification across `hub-entry`, `sokoban`, `reversi`, `undercover`, `board-games`, and `arcade-party`.

## Task Commits

No commits were created in this session. The plan was executed in an already dirty worktree and completed as working-tree changes only.

## Files Created/Modified

- `lib/party/undercover.js` - pure Undercover round engine and vote-resolution helpers
- `lib/party/manager.js` - Undercover room integration, dedicated route support, bot turns, and viewer-state serialization fixes
- `pages/undercover/[roomNo].js` - dedicated Undercover room page with clue, vote, roster, and result UX
- `styles/UndercoverRoom.module.css` - Undercover-specific room presentation
- `test-logic/undercover-logic.test.js` - logic coverage plus viewer-state contract protection
- `tests/undercover.spec.js` - browser smoke for one clue-and-vote loop on the dedicated room route
- `tests/hub-entry.spec.js` - exact-locator tightening for live homepage cards during consolidated validation
- `tests/sokoban.spec.js` - exact-text locator tightening for direct-launch copy during consolidated validation

## Decisions Made

- Undercover keeps its own dedicated room page instead of inflating the shared Werewolf/Avalon room shell with another divergent branch.
- Party-room serialization now exposes `viewer.ready` and `viewer.alive`, because active room pages need current-player state in addition to private role data.
- The final Wave 1 regression gate should run on a clean local split runtime when the default Docker app is stale or in-memory room counts have been polluted by prior smoke runs.

## Deviations from Plan

### Auto-fixed Issues

**1. Integration contract gap in party room viewer state**
- **Found during:** Task 2 consolidated Undercover/browser verification
- **Issue:** `pages/undercover/[roomNo].js` relies on `room.viewer.alive` and `room.viewer.ready`, but the party-room serializer only exposed role/side metadata.
- **Fix:** Added live seat state to the serialized `viewer` payload and locked it with a logic test.
- **Files modified:** `lib/party/manager.js`, `test-logic/undercover-logic.test.js`
- **Verification:** `node --test test-logic/backend-contract.test.js test-logic/client-network-contract.test.js test-logic/undercover-logic.test.js`, `npx playwright test tests/undercover.spec.js --workers=1`

**2. Consolidated Playwright smoke needed exact locators and longer Undercover stage waits**
- **Found during:** Task 2 consolidated Wave 1 verification
- **Issue:** live room cards and descriptive copy introduced duplicate text matches, and Undercover bot turns do not reach vote phase inside Playwright's default 5 s assertion window.
- **Fix:** tightened ambiguous selectors to exact headings/text where appropriate and made the Undercover vote-stage wait explicit.
- **Files modified:** `tests/hub-entry.spec.js`, `tests/sokoban.spec.js`, `tests/undercover.spec.js`
- **Verification:** `npx playwright test tests/hub-entry.spec.js tests/sokoban.spec.js tests/reversi.spec.js tests/undercover.spec.js tests/board-games.spec.js tests/arcade-party.spec.js --workers=1`

## Issues Encountered

- The long-running Dockerized app on `3100/3101` was serving stale code from the container image, so browser verification was moved to a clean local stack on `4100/4101`.
- Re-running room-creation smoke tests against a long-lived dev backend can exhaust the per-user open-room cap because room state is memory-resident; final verification was therefore run against a freshly restarted local stack.
