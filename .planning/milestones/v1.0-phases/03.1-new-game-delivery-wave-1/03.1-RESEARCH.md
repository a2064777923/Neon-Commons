# Phase 03.1: New Game Delivery Wave 1 - Research

**Researched:** 2026-04-22
**Domain:** Fast delivery of net-new playable games on top of the Phase 3 family hub, room-entry, and invite framework
**Confidence:** HIGH

<user_constraints>
## User Constraints

- New games now take priority over deeper expansion of existing Dou Dizhu / Werewolf / Avalon flows.
- This wave must put some new games directly into the product as usable experiences, not only placeholders.
- Unfinished approved titles must remain visible as greyed `即將推出` cards rather than disappearing.
- Approved candidate pool is fixed for this wave: `UNO 類`, `誰是臥底`, `你畫我猜`, `黑白棋`, `飛行棋`, `推箱子`, `保齡球`, `迷你賽車/碰碰車`.
- New titles must plug into the existing game-family arcade, room-entry, and invite-link model instead of bypassing it with ad hoc launch flows.
- Existing shipped gameplay remains the compatibility baseline; this wave must not regress Dou Dizhu, Werewolf, Avalon, Gomoku, or Chinese Checkers.

Derived project constraints from `AGENTS.md`, `.planning/PROJECT.md`, and prior phases:
- Keep the frontend/backend split intact. Backend work belongs in `backend/handlers/**` or shared backend modules, not `pages/api`.
- Reuse `lib/games/catalog.js`, `backend/handlers/hub.js`, `lib/shared/network-contract.js`, and the family hub already shipped in Phase 3.
- Reuse room-number, share-link, and guest/private-room flows for new multiplayer titles wherever applicable.
- Prefer incremental expansion over re-platforming. This phase should add titles through existing patterns, not invent a second product architecture.
</user_constraints>

<current_state_audit>
## Current State Audit

### What the codebase already gives us

- `lib/games/catalog.js` already models all approved titles as `coming-soon` entries inside the correct families.
- `backend/handlers/hub.js` plus `lib/admin/control-plane.js` already surface those cards through the homepage family bands.
- `pages/index.js` already renders live-vs-coming-soon cards from backend data instead of page-local constants.
- `backend/handlers/party/rooms/index.js` and `backend/handlers/board/rooms/index.js` already provide family-scoped create/list endpoints keyed by `gameKey`.
- `lib/board/manager.js` and `lib/party/manager.js` already own room lifecycle, AI fill, serialization, and result handling for multiplayer families.
- `pages/entry/[gameKey]/[roomNo].js`, invite-link resolution, and guest sync already exist for new multiplayer titles to reuse.

### What is still rigid

- `lib/board/manager.js` is explicitly hard-coded around only `gomoku` and `chinesecheckers`.
- `pages/board/[roomNo].js` is explicitly hard-coded around only Gomoku and Chinese Checkers themes, copy, move affordances, and board rendering.
- `lib/party/manager.js` is explicitly hard-coded around only `werewolf` and `avalon`.
- `pages/party/[roomNo].js` is explicitly hard-coded around only Werewolf and Avalon stage copy, action panels, and result labels.
- `pages/games/[gameKey].js` currently accepts only party/board families and does not support solo direct-play titles.
- The hub `liveFeed` in `backend/handlers/hub.js` only aggregates room-based families (`doudezhu`, `party`, `board`), so direct-launch solo titles need their own CTA strategy instead of relying on room feed presence.
- The Dou Dizhu/card stack is more coupled than board/party: template-backed room creation, persistent settlement assumptions, and room-page behavior are all specific to 斗地主.
</current_state_audit>

<recommended_live_subset>
## Recommended Playable-Now Subset

### 1. `推箱子` as the first direct-launch solo title

Why:
- It satisfies the user's explicit ask for more single-player variety.
- It does not require room lifecycle, invite links, or Socket.IO synchronization, so it can ship quickly.
- It fits the existing family arcade model as a `solo` card that launches directly into gameplay.

Implementation direction:
- Keep it under the existing `/games/` surface so the hub can route to `/games/sokoban`.
- Add a small self-contained solo game module (`lib/solo/**` and/or dedicated component files) rather than forcing it through room managers.
- Treat it as a direct-launch title in catalog/hub copy instead of a room-based one.

### 2. `黑白棋` as the first new room-based board title

Why:
- It is the cleanest extension of the current board architecture.
- It is still 2-player, turn-based, deterministic, and invite-friendly.
- It can reuse board-family room creation, room-number join, guest/private-room semantics, and a large part of the board UI shell.

Implementation direction:
- Extend board-family game metadata and handlers to accept `reversi`.
- Add Reversi-specific move validation, flipping logic, game-over detection, and AI heuristics in `lib/board/manager.js`.
- Reuse board APIs and room-entry contracts; only the actual board state and room rendering become game-specific.

### 3. `誰是臥底` as the first new party title

Why:
- It adds a distinctly different social loop from Werewolf/Avalon while still fitting the existing party-room/invite/voice rhythm.
- It avoids the heavy canvas/3D burden of `你畫我猜` / light-3D titles.
- It gives the wave one new real-time social game beyond classic deduction roles.

Implementation direction:
- Extend party-family metadata and handlers to accept `undercover`.
- Refactor binary Werewolf-vs-Avalon branching into a per-game dispatch structure where needed.
- Reuse party room pages, player rosters, phase timers, votes, and private invite flow, but render Undercover-specific rounds and accusation flow in dedicated UI branches or dedicated room pages.
</recommended_live_subset>

<architecture_implications>
## Architecture Implications

### Shared discovery and catalog work is still required

Even though Phase 3 already added placeholders, Phase 03.1 still needs a shared discovery pass to:
- promote selected titles from `coming-soon` to live routes
- keep unselected titles explicitly greyed `即將推出`
- ensure homepage CTA behavior works for both room-based and direct-launch titles
- keep family ordering and product copy coherent as new live titles appear

### Board family additions are lower risk than card or 3D additions

`lib/board/manager.js` and `pages/board/[roomNo].js` are rigid, but the rigidity is local and understandable: both files already separate Gomoku from Chinese Checkers with explicit branches. Extending that branch structure or extracting it into dispatch helpers for one more deterministic game is straightforward compared with inventing an entirely new runtime model.

### Party family additions need selective refactor, not replacement

`lib/party/manager.js` and `pages/party/[roomNo].js` will need refactoring because they assume exactly two games, but the surrounding architecture is still the right home for Undercover:
- room creation/list/join already exists
- private/public visibility and guest/private invite rules already exist
- player feed, timers, votes, and result overlays already exist

The right move is to introduce a third party game through targeted dispatch tables and game-specific panels, not to create a second party-runtime stack.

### Solo launch should stay outside room semantics

A solo title should not be forced into fake room numbers just to fit the rest of the platform. The family arcade already supports non-room titles as long as catalog metadata and CTA routing are explicit. `推箱子` should therefore launch directly from its family card and not pretend to be an invite/join game.
</architecture_implications>

Use **4 plans**:

1. Shared new-game launch contract and family discovery promotion
2. Sokoban solo implementation
3. Reversi board-room implementation
4. Undercover party-room implementation plus wave validation

Recommended execution order:
- Plan 01 first, because it promotes the live subset and establishes the routing/discovery contract for the other three plans.
- Plans 02, 03, and 04 can follow after Plan 01, but only in parallel where write sets do not overlap.
- If plan execution needs strict disjointness, run Sokoban and one multiplayer title in parallel, then finish the remaining multiplayer title and consolidated smoke coverage after.

## Open Questions

1. **Should this wave attempt a new card-family game now?**
   - What we know: the existing card stack is 斗地主-specific and economically coupled.
   - Recommendation: no. Keep `UNO 類` visible as `即將推出` and avoid turning this phase into a parallel card-engine rewrite.

2. **Should Sokoban reuse room/invite infrastructure anyway for consistency?**
   - What we know: the user wants more single-player variety, and solo titles do not benefit from room-number friction.
   - Recommendation: no. Launch it directly from `/games/sokoban` and keep invite/room semantics for multiplayer titles only.

3. **Should Undercover reuse the generic `pages/party/[roomNo].js` room shell or get its own dedicated room page?**
   - What we know: current party page is strongly binary around Werewolf vs Avalon.
   - Recommendation: allow the plan to choose either:
     - refactor `pages/party/[roomNo].js` into a 3-game shell if the write set stays understandable, or
     - create a dedicated `pages/undercover/[roomNo].js` room page if that isolates complexity better.
   - The same trade-off applies to Reversi vs the current generic board room shell.
