# Phase 03.1: New Game Delivery Wave 1 - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the first real wave of net-new games through the hub and room-entry framework built in Phase 3, before spending the next major phase on deeper expansion of existing shipped families. This phase should get fresh playable surface into the arcade quickly, while allowing unfinished approved titles to remain visible as `即將推出` rather than blocking the whole wave.

</domain>

<decisions>
## Implementation Decisions

### Phase Priority
- **D-01:** New games now take priority over deeper expansion of existing Dou Dizhu, Werewolf, and Avalon gameplay.
- **D-02:** This priority change is handled as inserted Phase `03.1`, which runs before the pre-existing Phase 4 roadmap work.

### Approved New-Game Pool
- **D-03:** The approved candidate pool for this wave is `UNO 類`, `誰是臥底`, `你畫我猜`, `黑白棋`, `飛行棋`, `推箱子`, `保齡球`, and `迷你賽車/碰碰車`.
- **D-04:** This wave should ship at least part of that pool as directly playable games, not only as hub placeholders.
- **D-05:** Titles from the approved pool that are not finished in time should remain visible in the hub as greyed `即將推出`.

### Delivery Shape
- **D-06:** Prefer lower-integration-cost titles first so the arcade gains more playable variety quickly.
- **D-07:** Heavier real-time drawing or light-3D titles may ship in this wave only if feasible without slowing down easier wins; otherwise they stay as `即將推出`.
- **D-08:** The planner may choose the exact playable-now subset by feasibility, but it must preserve the user's approved title pool and the "some playable now, some coming soon" delivery model.

### Product and Entry Constraints
- **D-09:** New titles must plug into the existing game-family hub structure instead of bypassing it with one-off launch flows.
- **D-10:** New multiplayer titles should reuse the room-number, invite-link, and guest/private-room rules already established in Phase 3 wherever applicable.
- **D-11:** Existing shipped games remain the compatibility baseline and must not regress while new titles are added.

### the agent's Discretion
- Exact split between `playable now` and `即將推出` across the approved title pool.
- Exact family placement for each new title where the family fit is flexible.
- Whether a title should launch as single-player first, private-room multiplayer first, or both, as long as the hub and entry model stay coherent.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `lib/games/catalog.js`: Already models shipped and upcoming titles, family grouping, route metadata, share-link support, and audience tags.
- `pages/index.js`: Already renders the family-based homepage and can surface additional new games without another IA rewrite.
- `backend/handlers/hub.js`: Already emits the discovery payload the homepage consumes.
- `backend/handlers/room-entry/*`: Already centralizes room-number, invite-link, guest-entry, and post-match claim behavior for multiplayer titles.

### Established Patterns
- New titles can appear in two product states: directly playable now or greyed `即將推出`.
- Per-game dedicated entry pages remain acceptable for deeper room setup; the homepage stays the shared family discovery surface.
- Capability visibility and room-entry rules are already centralized and should be reused instead of creating ad hoc flows per title.

### Integration Points
- New title metadata should land in `lib/games/catalog.js` and flow outward through `backend/handlers/hub.js` and `pages/index.js`.
- New playable games will need family-appropriate handlers/pages/managers that attach to the same room/discovery framework where multiplayer behavior is required.
- Single-player titles can still benefit from the same catalog/hub surfacing even if they do not require the full room-entry path.

</code_context>
